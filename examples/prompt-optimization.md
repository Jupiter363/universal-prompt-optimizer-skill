# Prompt Optimization 示例库

v0.2.0 采用两阶段流水线：**追问澄清 → 提示词标准化**，标准化后默认执行。每个示例标注了追问轮次和格式（完整流水线 / 标准化结果）。

---

## 0. 完整流水线演示

### 示例：Debug — 登录间歇性故障

**用户输入：**

> 帮我修一下登录功能，用户说有时候登录不了

**阶段一 · 追问（第 1 轮）：**

> 在继续之前需要确认几个信息（不清楚的可以答"不确定"）：
> 1. 具体的异常表现——登录时报错/白屏/没反应/自动退出？
> 2. 是所有用户还是部分用户受影响？

**用户回答：**

> 所有用户，点击登录按钮后没反应，无报错。最近改过 Token 刷新逻辑

**信息熵判断：** 目标明确（按钮无响应 + Token 刷新变更），范围已定，安全无风险 → 停止追问。

**阶段二 · 标准化 → 默认执行：**

> **追问汇总：**
> - 问：异常表现？ → 答：点击登录后无反应，无报错
> - 问：影响范围？ → 答：所有用户
> - 问：最近改动？ → 答：Token 刷新逻辑
>
> **标准化 Prompt（7 段结构）：**
>
> | 维度 | 内容 |
> |------|------|
> | **角色** | 前端调试专家，擅长浏览器端异步逻辑和网络请求 |
> | **任务** | 排查并修复登录按钮点击后无响应的问题 |
> | **背景** | 所有用户有概率遇到，点击登录后页面无反应、无报错。最近改动 Token 刷新逻辑，疑似相关 |
> | **目标** | 定位根因并修复，所有用户点击登录均正常发起请求 |
> | **约束** | 不得重构认证模块；不得修改后端登录接口；修改前 git diff 确认 Token 刷新逻辑变更 |
> | **步骤** | 1. 定位登录按钮点击事件绑定 2. 检查是否被 Token 刷新异常中断 3. Network 面板确认请求是否发出 4. 确认根因后最小修复，验证两流程互不干扰 |
> | **输出** | 问题根因、修复方式、涉及文件、验证步骤 |
>
> **未确认信息：** 网络请求是否正常发出（建议先打开 Network 面板确认）

### 示例 1.2：完整流水线 — API 接口

**用户输入：** 帮我写个用户注册接口

**阶段一 · 追问（1 轮）：**

> 需要确认：
> 1. 需要哪些注册字段——仅邮箱+密码，还是需要手机号、用户名、验证码？
> 2. 技术栈和后端框架是什么？

**用户回答：** Node.js + Express，邮箱+密码+用户名即可，不需要手机验证

**标准化 Prompt：**

| 维度 | 内容 |
|------|------|
| **角色** | 后端开发工程师，熟悉 Node.js/Express |
| **任务** | 实现用户注册 API 接口（POST /api/auth/register） |

| **约束** | 密码必须哈希存储（bcrypt）；邮箱格式校验；用户名 2-20 字符；不实现邮件验证（后续迭代）；遵循项目已有路由和中间件模式 |
| **步骤** | 1. 检查项目现有路由结构和用户模型 2. 创建注册路由和控制器 3. 实现字段校验 4. 密码 bcrypt 哈希 5. 写入数据库并返回成功响应 |
| **输出** | 涉及文件、关键代码说明、接口请求/响应示例、验证方式 |
| **未确认** | 数据库类型、是否需要返回 JWT Token |

## 总结

以上示例的共同特征：

1. **先追问再标准化**：模糊输入不直接优化，缺失关键信息时进入追问，方向明确后停止
2. **固定 7 段结构**：角色/任务/背景/目标/约束/步骤/输出——每个维度有内容或显式标注未确认
3. **去口语化**：原始"帮我修 bug""帮我加个功能"等表达，一律转为精确的任务描述
4. **默认执行**：标准化 Prompt 输出后直接执行，不额外确认
