# 优化前后对比示例

v0.2.0 采用两阶段流水线：**追问澄清 → 提示词标准化**，标准化后默认执行。

本文档第 0 节展示完整流水线，第 1~8 节展示各任务类型的标准化前后对比。

---

## 0. 完整流水线（Debug 类）

**用户输入：**
> 帮我修一下登录功能，用户说有时候登录不了

**阶段一 · 追问（1 轮即停）：**
> 在继续之前需要确认几个信息（不清楚的可以答"不确定"）：
> 1. 具体的异常表现——登录时报错/白屏/没反应/自动退出？
> 2. 是所有用户还是部分用户受影响？

**用户回答：**
> 所有用户，点击登录按钮后没反应，无报错。最近改过 Token 刷新逻辑

**信息熵判断：** 目标明确（按钮无响应 + Token 刷新变更），范围已定，安全无风险 → 停止追问。

**阶段二 · 标准化 → 默认执行：**

> **追问汇总：**
> - 问：异常表现？ → 答：点击登录后无反应，无报错
> - 问：影响范围？ → 答：所有用户
> - 问：最近改动？ → 答：Token 刷新逻辑
>
> **标准化 Prompt（7 段结构）：**
>
> 角色 · 任务 · 背景 · 目标 · 约束 · 步骤 · 输出格式
>
> | 维度 | 内容 |
> |------|------|
> | **角色** | 前端调试专家，擅长浏览器端异步逻辑和网络请求 |
> | **任务** | 排查并修复登录按钮点击后无响应的问题 |
> | **背景** | 所有用户有概率遇到，点击登录按钮后页面无反应、无报错。最近改动 Token 刷新逻辑，疑似相关 |
> | **目标** | 定位根因并修复，所有用户点击登录均正常发起请求 |
> | **约束** | 不得重构认证模块；不得修改后端登录接口；修改前 git diff 确认 Token 刷新逻辑变更 |
> | **步骤** | 1. 定位登录按钮点击事件绑定 2. 检查是否被 Token 刷新异常中断 3. Network 面板确认请求是否发出 4. 确认根因后最小修复，验证两流程互不干扰 |
> | **输出** | 问题根因、修复方式、涉及文件、验证步骤 |
>
> **未确认信息：** 网络请求是否正常发出（建议先打开 Network 面板确认）

## 1. 代码开发类

**原始输入：** 帮我加个搜索功能

**追问要点：** 搜索范围（全文/标题/字段）、数据源、分页排序需求、UI 交互方式

**标准化 Prompt：**

| 维度 | 内容 |
|------|------|
| **角色** | 全栈开发工程师 |
| **任务** | 为现有项目添加搜索功能 |
| **背景** | [未确认：需确认搜索的具体页面和模块] |
| **目标** | 用户可通过关键词搜索目标内容，结果支持分页展示 |
| **约束** | 遵循已有架构模式；只添加搜索相关代码，不修改无关模块；处理空结果和错误状态 |
| **步骤** | 1. 检查项目现有结构和路由配置 2. 实现搜索接口/查询逻辑 3. 实现搜索输入和结果展示 UI 4. 添加防抖和基本性能优化 5. 处理空结果和网络错误状态 |
| **输出** | 涉及文件、关键实现说明、验证方式 |
| **未确认** | 搜索范围、数据源、分页需求、搜索触发方式 |

## 2. Debug 类

**原始输入：** 接口报错了，帮我看看

**追问要点：** 具体接口路径、报错码和消息、请求参数、复现步骤

**标准化 Prompt：**

| 维度 | 内容 |
|------|------|
| **角色** | 后端调试专家 |
| **任务** | 排查接口报错问题 |
| **背景** | [未确认：接口路径和报错内容未提供] |
| **目标** | 定位报错根因并修复 |
| **约束** | 先定位问题再动手；只做最小修复；不修改无关接口；保持接口签名兼容 |
| **步骤** | 1. 获取具体报错信息（错误码、消息、堆栈） 2. 定位接口代码 3. 检查请求参数、响应格式 4. 分析可能原因（参数校验/数据库异常/权限/第三方服务） 5. 确认根因后最小修复 |
| **输出** | 错误原因分析、修复方案、涉及文件、验证方式 |
| **未确认** | 接口路径、报错详情、请求参数、复现步骤 |

## 3. 重构类

**原始输入：** 帮我重构一下这段代码，太乱了

**追问要点：** 具体文件/模块路径、重构原因（太长/重复/逻辑混乱）、现有测试情况

**标准化 Prompt：**

| 维度 | 内容 |
|------|------|
| **角色** | 代码重构专家 |
| **任务** | 在保持行为不变的前提下重构指定代码 |
| **背景** | 用户认为代码结构混乱，难以维护 |
| **目标** | 改进代码可读性和可维护性，所有现有测试通过 |
| **约束** | 不得改变外部行为；不得修改公共接口签名（除非确认无调用者）；重构范围仅限于用户指定区域；确保现有测试通过 |
| **步骤** | 1. 阅读并理解当前逻辑 2. 识别具体问题（函数过长/命名不清/重复/职责混乱/嵌套过深） 3. 列出重构计划 4. 逐步重构，每步后确认行为不变 |
| **输出** | 重构前后关键差异、涉及文件、验证建议 |
| **未确认** | 具体文件或代码段路径、已有测试情况 |

## 总结

以上所有示例的共同特征：

1. **先追问再标准化**：模糊输入不会直接优化，缺失关键信息时进入追问，方向明确后停止
2. **固定 7 段结构**：角色/任务/背景/目标/约束/步骤/输出——每个维度有内容或显式标注未确认，不留空洞
3. **去口语化**：原始"帮我修 bug""帮我加个功能"等表达，一律转为精确的任务描述
4. **默认执行**：标准化 Prompt 输出后直接执行，不额外确认
