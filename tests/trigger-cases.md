# 触发测试用例

本文档用于验证 Universal Prompt Optimizer Skill 是否能被正确触发。

测试方法：将以下输入提供给支持 Skill 的 Agent 工具（Claude Code / Codex 等），观察是否会触发 `universal-prompt-optimizer` Skill。

## 五、追问模式测试（v0.2.0 新增）

| 编号 | 输入 | 用户模拟回答 | 预期行为 |
|------|------|------------|---------|

## 七、回归测试检查清单

每次修改 `SKILL.md` 后，应检查以下项目：

**触发类：**
- [ ] 显式触发用例 E-01 到 E-17 是否仍然能触发
- [ ] 隐式触发用例 I-01 到 I-15 的触发概率是否下降
- [ ] 不应触发用例 N-01 到 N-08 是否被误触发

**追问类：**
- [ ] 缺失关键信息时是否进入追问模式
- [ ] 每轮问题数是否控制在 1-3 个
- [ ] 信息熵达标时是否立即停止追问
- [ ] 3 轮硬上限是否生效
- [ ] 用户说"不知道"时是否接受并继续

**优化质量类：**
- [ ] 优化后的 Prompt 是否保留了用户原意
- [ ] 优化后的 Prompt 是否没有编造缺失信息
- [ ] 优化后的 Prompt 是否结构清晰、可执行
- [ ] 追问汇总是否正确记录了问答内容
- [ ] 未确认信息是否显式标注

**输出模式类：**
- [ ] 默认是否在标准化后直接执行（不额外确认）
- [ ] "只优化不执行"是否跳过执行环节
- [ ] 极简模式是否跳过追问且不执行

---

## 已知限制

1. 触发概率受主模型影响，不同模型对同一输入的匹配能力可能不同
2. 模糊输入触发是"概率性"的，不保证 100% 触发
3. `description` 的长度和覆盖度直接影响触发率，可能需要多轮迭代优化
4. 不同 Agent 工具对 Skill 触发机制的理解不同，需要在实际环境中测试
5. 信息熵判断是 LLM 主观评估，不同模型对"信息是否足够"的判断可能存在差异
